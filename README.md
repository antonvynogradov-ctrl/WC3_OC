## 1. Процедури тиражування та міграції віртуальних середовищ

### 1.1 Метод повного клонування (Full Clone)
Для створення ідентичного робочого середовища було застосовано інструментарій клонування. Це критично важливо для тестування мережевої взаємодії без потреби в повторній інсталяції та базовому налаштуванні ОС. 

**Технічні особливості процесу:**
* **Перегенерація MAC-адрес:** Під час клонування було обрано політику створення нових фізичних адрес для мережевих адаптерів. Це обов'язкова умова для уникнення конфліктів у канальному рівні мережі (L2).
* **Тип клону:** Обрано "Full Clone", що гарантує повну автономність нової машини від оригіналу, копіюючи всі дані віртуального жорсткого диска.

### 1.2 Експорт та перенесення систем (Appliance Export)
Для забезпечення мобільності проєкту було виконано процедуру експорту ВМ у формат відкритої віртуалізації (OVA). Цей процес перетворює складну структуру файлів ВМ у єдиний стиснутий архів, який містить опис апаратної конфігурації (CPU, RAM, порти) та образ диска. Це дозволяє розгорнути систему на будь-якому іншому гіпервізорі (наприклад, VMWare).

## 2. Теоретичний аналіз мережевих інтерфейсів гіпервізора

Під час виконання лабораторної роботи було детально вивчено та практично випробувано чотири режими роботи мережі:

1. **Трансляція адрес (NAT):** Це найбільш захищений режим за замовчуванням. ВМ отримує доступ до зовнішніх ресурсів (Internet), але залишається невидимою для зовнішніх запитів.
2. **Мережевий міст (Bridged Adapter):** Використовується, коли ВМ має функціонувати як окремий сервер у реальній локальній мережі. Машина отримує IP безпосередньо від DHCP-сервера роутера.
3. **Віртуальний адаптер хоста (Host-Only):** Створює замкнений контур між основною ОС (Windows) та віртуальною (Linux). Це ідеально для випадків, коли потрібно керувати ВМ через SSH з основної системи, не випускаючи її в інтернет.
4. **Внутрішня мережа (Internal Network):** Повністю ізольована віртуальна шина даних. Тільки машини всередині одного гіпервізора бачать одна одну. Це дозволяє моделювати атаки або складні зв'язки без ризику для основної мережі.

## 3. Практичне розгортання та діагностика локальної мережі

### Системні утиліти та моніторинг:
Для налаштування та перевірки зв'язку було використано наступний стек команд:
* `ip a` (заміна застарілої `ifconfig`) — для ідентифікації призначених адрес.
* `ping -c 4 [IP]` — для перевірки цілісності каналу та затримок пакетів (RTT).
* `sudo ip link set [interface] up` — ініціалізація мережевого адаптера після зміни конфігурації.
* `netcat` (`nc`) — для створення TCP/UDP потоків даних між вузлами.

### Перевірка зовнішнього зв'язку (Internet Connectivity):
На обох машинах підтверджено можливість роботи з медіаконтентом (YouTube) через браузер Firefox, що свідчить про коректне налаштування DNS та маршрутизації через NAT.

## 4. Інтеграція та синхронізація даних (Host ↔ Guest)

Важливою частиною роботи була організація обміну інформацією між Windows (Host) та Linux (Guest).

1. **Спільні папки (Shared Folders):** Налаштовано монтування директорії Windows у файлову систему Linux. Це дозволяє маніпулювати великими обсягами даних без потреби у флешках чи мережевих дисках.
2. **Передача медіафайлів:** Було успішно скопійовано аудіофайли з основної системи на робочі столи обох віртуальних машин.

## ✅ Загальний висновок

Виконання даного кейсу дозволило на практиці зрозуміти механіку роботи гіпервізорів з мережевими рівнями. Встановлено, що правильна конфігурація MAC-адрес при клонуванні є критичною для стабільності мережі. Налаштування внутрішньої мережі (`Internal Network`) у поєднанні з NAT дозволило одночасно мати доступ до глобальної мережі та підтримувати ізольований канал зв'язку між двома системами. Отриманий досвід є фундаментальним для подальшого вивчення системного адміністрування та кібербезпеки.
